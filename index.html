Проверка подлинности носителя JWT (JSON Web Token) обычно используется для API. Хотя он работает аналогично аутентификации cookie, поставщик удостоверений выдает JWT или токены при успешной аутентификации. Затем эти маркеры можно отправлять на другие серверы для проверки подлинности, в отличие от файлов cookie, которые отправляются только в выданный домен.

Проверка подлинности носителя JWT обеспечивает:

  • Аутентификация: При использовании JwtBearerHandler маркеры доступа необходимы для аутентификации. JwtBearerHandler проверяет токен и извлекает идентификацию пользователя из утверждений.
  • Авторизация: Маркеры носителя обеспечивают авторизацию, предоставляя коллекцию утверждений, представляющих разрешения пользователя или приложения, подобно cookie.
  • Делегированная авторизация: Если маркер доступа используется для делегирования полномочий между API вместо маркера доступа на уровне приложения, этот процесс называется делегированной авторизацией.

Введение в аутентификацию JWT с помощью носителя см. в JSON веб-токенах. Просмотр или скачивание примера кода.

Типы токенов

Существует множество типов токенов и форматов. Создание собственных маркеров доступа или маркеров идентификатора не рекомендуется, за исключением целей тестирования. Самостоятельно созданные токены, которые не соответствуют установленным стандартам, могут вызвать проблемы безопасности и совместимости.

Токены доступа

  • Строки, используемые клиентским приложением для выполнения запросов к серверу, реализующему API.
  • Могут отличаться в формате. Различные API могут использовать разные форматы для токенов.
  • Можно зашифровать.
  • Веб-клиент или приложение пользовательского интерфейса с маркером доступа никогда не должно читать или интерпретировать маркер доступа.
  • Предназначены исключительно для выполнения запросов к API.
  • Обычно отправляются в API в качестве маркера доступа в заголовке запроса авторизации.

Маркеры доступа приложений и делегированные маркеры доступа

Маркеры доступа могут быть маркерами доступа приложений или маркерами делегированного доступа. Токены содержат разные данные и управляются и хранятся по-разному.

  • Маркер доступа к приложению обычно хранится один раз в приложении до истечения срока его действия.
  • Маркер делегированного доступа хранится для каждого пользователя либо в cookie, либо в защищенном кеше сервера.
  • Рекомендуется использовать делегированные токены доступа пользователей при каждом участии пользователя.
  • Подчиненные API могут запрашивать делегированный маркер доступа пользователя для имени прошедшего проверку подлинности пользователя.

Совет

Используйте безопасное хранилище для маркеров, чтобы избежать утечек данных и повысить защиту приложения.
